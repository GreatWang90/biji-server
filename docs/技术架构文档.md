## 2. 技术栈

| 层次 | 技术 | 说明 |
|------|------|------|
| 前端 | HTML5 + CSS3 + ES6 | 单文件应用，无框架依赖 |
| 后端 | Node.js | 运行时环境 |
| 通信 | WebSocket (ws@8.16.0) | 实时双向通信 |
| HTTP | Node.js http 模块 | 内置模块，静态文件服务 |
| 存储 | JSON 文件 | data.json 本地持久化 |
| 部署 | Render.com / 本地 | 支持云平台和局域网 |

## 4. 服务端模块划分

### 4.1 数据存储模块
- `loadData()` - 启动时从 data.json 加载数据
- `saveData()` - 将内存数据写入 data.json（使用 JSON replacer 排除 `_timer` 等不可序列化字段）

### 4.2 连接管理模块
- `clients` Map - 维护 WebSocket 连接与用户信息的映射
- `broadcast()` - 向房间内所有玩家广播消息
- `sendTo()` - 向单个玩家发送消息
- `broadcastRoomState()` - 广播房间状态（过滤敏感信息）

### 4.3 牌组逻辑模块
- `createDeck(size)` - 创建52/54张牌组
- `shuffle(arr)` - Fisher-Yates 洗牌算法
- `cardValue(card)` / `cardSuitValue(card)` - 牌面/花色数值映射

### 4.4 业务处理模块
- `handleRegister()` - 注册验证与用户创建
- `handleLogin()` - 登录验证
- `handleCreateRoom()` - 创建房间并初始化
- `handleJoinRoom()` - 加入房间（含重连支持）
- `handleLeaveRoom()` - 离开房间与房主转移
- `handleToggleReady()` - 准备状态切换与自动开始检测
- `startGame()` - 服务端发牌与游戏状态初始化
- `handleSubmitArrange()` - 接收摆牌并检测全员完成
- `forceFinishArrange()` - 超时强制摆牌
- `finishArrange()` - 广播所有玩家的摆牌结果
- `handleStartNextRound()` - 金币同步与房间状态重置（first-call-only）

## 5. 前端模块划分

### 5.1 页面管理
- 6个页面（auth/lobby/room/game/result/rules）通过 CSS `display` 切换
- `showPage(id)` 统一控制页面可见性

### 5.2 WebSocket 管理
- 自动连接，断线后每2秒重连
- 连接状态指示器（右上角绿色/红色）
- 消息路由：根据 `type` 字段分发处理

### 5.3 游戏渲染
- `renderCard(card)` - SVG方式渲染扑克牌
- `renderArrangeArea()` - 渲染可交互的摆牌区域
- `renderArrangeAreaReadonly()` - 确认后的只读摆牌展示
- `handleAllArranged()` - 比牌结果动画渲染
- 6个SVG卡通头像内联定义

### 5.4 游戏逻辑（客户端）
- `getHandType(cards)` - 牌型判定（豹子/同花顺/同花/顺子/对子/散牌）
- `compareHands(a, b)` - 两手牌比较
- `autoArrange(cards)` - AI智能摆牌（穷举C(9,3)*C(6,3)=840种组合）
- 结算计算（逐墩比较、通杀判定、金币变化）

```javascript
{
```

## 7. 关键技术决策

| 决策 | 原因 |
|------|------|
| 单 HTML 文件 | 简化部署，无需构建工具，一个文件即完整前端 |
| 服务端发牌 | 防止客户端篡改牌面数据，核心反作弊手段 |
| JSON 文件存储 | 轻量级，无需数据库，适合小型项目 |
| WebSocket | 实时性要求高，HTTP 轮询不适合游戏场景 |
| SVG 头像内联 | 无需外部图片资源，减少 HTTP 请求 |
| 客户端比牌 | 减轻服务端计算压力，比牌逻辑可复用于单机版 |
| Fisher-Yates 洗牌 | 标准均匀随机洗牌算法，公平性保证 |

## 8. 已知限制

- 密码明文存储（适用于娱乐项目，生产环境需 hash 处理）
- 无身份令牌机制（断线重连需重新登录）
- JSON 文件存储不适合高并发（适合小型部署）
- 客户端比牌结算后上报金币，理论上可被篡改（可通过服务端结算加固）
